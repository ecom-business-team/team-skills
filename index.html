Ecom Business Team — Skills Marketplace

A shared library of Claude Code skills for the team. No terminal required. Works in the VSCode extension.

Start Here — First-time setup

You need Claude Code installed (VSCode extension is fine). Then two steps:

  1. Open a Claude Code chat. Paste the prompt below exactly as-is and send it. Claude will do the file setup for you (about 15 seconds).
  2. When Claude confirms it's done, close the Claude Code chat and open a new one. Then type /team-skills-browse to install the plugin for your role.
Please install the ecom-business-team team-skills toolkit. Clone https://github.com/ecom-business-team/team-skills.git to ~/.cache/team-skills, then copy every skill folder inside plugins/general/skills/ (there should be five: new-workflow, onboard, team-skills-add, team-skills-browse, team-skills-update) into ~/.claude/skills/. Write "general" into a .team-skills-source file inside each copied skill folder. When you're done, tell me it's ready and that I can run /team-skills-browse next.

After setup, these three commands are how you interact with the marketplace:

  1. /team-skills-browse — see what plugins are available and install what you need
  2. /team-skills-update — refresh every team-skill you have to the latest version
  3. /team-skills-add — share a skill you built with the team (opens a pull request for you)

Quality review

What breaks?
Checklist too strict: teammate can't get through the fork/branch flow and gives up. Mitigation: /team-skills-add tells them the exact fix and lets them re-run.
Silent merge conflict: two teammates submit the same skill name. Mitigation: PR review catches it before merge.
Stale local clone: teammate installs before /team-skills-update. Mitigation: /team-skills-browse always fetches live from GitHub, so the catalog is never stale.
Who knows?
PR opened: plugin owner (GitHub notification). If no owner assigned yet, PR sits — need to check the "assign owners" open decision.
Skill fails checklist: teammate sees it immediately in /team-skills-add.
Skill breaks after merge: next teammate to install it will report — no automated regression testing yet (accepted risk given scope).
What's the fallback?
Skill installation fails: teammate can manually run copies each skill folder into ~/.claude/skills/ — /team-skills-browse is a convenience layer, not a dependency.
Skill submission fails: teammate can open a PR by hand through GitHub UI — /team-skills-add is a convenience wrapper over standard git.
Bad skill merged: revert PR + /team-skills-update propagates the removal.
